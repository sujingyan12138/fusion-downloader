# 项目协作规则

## 项目定位

- 本项目是 Windows 本地 Tkinter GUI 下载器，平台能力通过 `downloaders/` 中的独立模块接入，由 `services/task_runner.py` 统一调度，`app.py` 只负责界面、输入校验和后台任务生命周期。
- 下载产物统一写入应用同级的 `下载结果/`。新增平台不得另设隐蔽的默认输出根目录。
- 平台专属解析、请求头、下载、命名和异常应留在对应平台模块；跨平台任务编排放在 `services/`，不要继续膨胀 `app.py`。

## 开发环境与常用命令

- 默认环境：Windows + PowerShell；项目虚拟环境为 `.venv`。
- 启动 GUI：`\.venv\Scripts\python.exe app.py`
- 安装依赖：`\.venv\Scripts\python.exe -m pip install -r requirements.txt`
- 运行测试：`\.venv\Scripts\python.exe -m unittest discover -s tests -v`
- 语法检查：`\.venv\Scripts\python.exe -m compileall -q app.py downloaders services tests`
- 打包：`\.\build_exe.bat`；打包逻辑以 `build_exe.ps1` 为准。
- 运行会产生下载文件的测试时使用单独的临时目录；不要污染或清空用户的 `下载结果/`。

## 代码约定

- 维持 Python 类型标注和 `from __future__ import annotations` 风格；面向 GUI 的错误消息使用清晰中文，平台模块提供专属 `RuntimeError` 子类。
- 网络请求必须设置超时、明确必要的 `User-Agent`/`Referer`，并对重定向、空响应、受限清晰度和解析结构变化给出可理解的失败信息。
- 文件名必须经过 Windows 安全化处理，并包含平台、标题或作者、稳定内容 ID；同名文件不得静默覆盖。
- GUI 耗时工作必须在后台线程运行；Tkinter 控件更新继续通过现有队列回到主线程。
- 不得记录或提交 Cookie、token、浏览器账号数据、完整敏感请求头或登录态目录内容。
- 登录 Cookie 只允许发送给对应平台的页面/API 域名；已签名的 CDN 媒体 URL 不得附带账号 Cookie，避免跨域泄露。
- 修改依赖时同步更新 `requirements.txt`、README 使用说明和 PyInstaller 打包路径；外部工具应在运行前检测并给出明确安装/打包提示。

## 媒体下载验收标准

- “最高质量”指在当前账号权限和源站可用范围内，选择质量排序最高的视频流与最佳音频流；不得为了单文件直链而主动降级画质。
- 当最高质量为分离音视频流时，必须无重编码合并，并验证最终文件同时包含视频和音频；缺少合并工具时不得把无声视频误报为成功。
- 下载过程使用临时文件时，成功后清理临时文件；失败时保留可诊断信息但不要留下被误认作完成品的文件。
- 任务报告至少包含输出路径、内容 ID、标题、所选格式/清晰度以及失败列表；批量任务中单条失败不应隐藏其他条目的结果。
- 新增平台至少覆盖：链接提取/去重、非法输入、质量选择、输出命名、调度分支和失败聚合的自动化测试；涉及真实站点的验证需另行记录链接、时间、账号边界和实际媒体信息。
- 完成前运行单元测试、语法检查，并至少手动回归一次受影响的 GUI 任务构建或调度路径。

## 项目级记忆

- 复杂、多阶段任务使用根目录的 `task_plan.md`、`findings.md`、`progress.md`；已有内容应合并更新，不得直接覆盖。
- `task_plan.md` 只保存目标、阶段、决策和错误；外部网页或接口内容仅写入 `findings.md`；执行和验证日志写入 `progress.md`。
- 每完成阶段、遇到错误或改变技术方案后更新规划文件；恢复任务时先重读三份文件并核对 Git 状态。
- 项目实现细节和临时状态不得写入跨项目经验总结。只有经当前证据验证、可跨项目复用的经验才按用户指定流程评估写回。

## 变更边界

- 保留用户已有修改；开始前检查 `git status`，不要使用 `git reset --hard`、`git checkout --` 等破坏性命令清理工作区。
- 不提交构建产物、下载媒体、浏览器登录态、缓存或临时文件。
- 未经明确要求，不创建提交、不推送远端、不删除用户下载内容。
